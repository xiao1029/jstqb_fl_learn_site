2.2 テストレベルとテストタイプ

2.2.1

テストレベル

K2・理解
この項の学習目的さまざまなテストレベルを区別する。

各テストレベルはテスト対象・テスト目的・テストベース・想定する欠陥や故障・アプローチと責務によって区別される。

  • コンポーネントテスト(ユニットテスト) — コンポーネント単体を、通常開発担当者が開発環境で実施
  • コンポーネント統合テスト — コンポーネント間のインターフェースと相互処理に焦点。統合戦略(ボトムアップ/トップダウン/ビッグバン)に依存
  • システムテスト — システム全体の機能・非機能の振る舞いに焦点。独立したテストチームが行うこともある
  • システム統合テスト — テスト対象システムと他システム・外部サービスとのインターフェースに焦点
  • 受け入れテスト — 妥当性確認とデプロイ準備の実証が目的。UAT・運用受け入れ・契約/規制受け入れ・アルファ/ベータテストなどがある
2.2.2

テストタイプ

K2・理解
この項の学習目的さまざまなテストタイプを区別する。

これら4つのテストタイプは焦点こそ異なるが、すべてのテストレベルに適用できる点が重要。

  • 機能テスト — 「何をすべきか」を評価。機能完全性・正確性・適切性をチェック
  • 非機能テスト — 「どのようにうまく振る舞うか」を評価。性能効率性・互換性・使用性・信頼性・セキュリティ・保守性・移植性(ISO/IEC 25010)
  • ブラックボックステスト — 仕様に基づきテスト対象の外観からテストを導出
  • ホワイトボックステスト — 内部構造(コード・アーキテクチャー等)からテストを導出
2.2.3

確認テストとリグレッションテスト

K2・理解
この項の学習目的確認テストとリグレッションテストを区別する。

確認テストは「元の欠陥が正しく修正されたか」を確認するテスト。リグレッションテストは「その変更が他の部分に悪影響を及ぼしていないか」を確認するテストで、すでに合格していたテストを再実行する。リグレッションテストは繰り返し実行されるため自動化の有力候補となる。
